# Терминология  

## Transformers

### Что такое Transformers?  
- Ответ:  
  **Transformers** — это архитектура нейронных сетей, которая была представленаа в 2017 году в статье ["Attention is All You Need"](https://arxiv.org/abs/1706.03762). Она использует механизм внимания (self-attention) для обработки последовательностей данных, что позволяет моделям эффективно учитывать контекст и зависимости между элементами входных данных. Transformers стали основой для многих современных моделей **обработки естественного языка** (NLP - Natural Language Processing ), таких как BERT, GPT и T5, и широко применяются в задачах машинного перевода, генерации текста и других областях ИИ.

### Классификация моделей

LLM могут иметь различную архитектуру в зависимости от их целей, вычислительных ресурсов и задач обучения. Также их можно разделить по архитектурам:

- **(На базе энкодера) BERT** (Bidirectional Encoder Representations from Transformers) — модель, которая смотрит на текст в обоих направлениях для лучшего понимания контекста.  
- **(На базе декодера) GPT** (Generative Pre-trained Transformer) — модель, предназначенная в первую очередь для генерации текста.  
- **(Энкодер + декодер) T5** (Text-to-Text Transfer Transformer) — модель, которая рассматривает все задачи обработки естественного языка как преобразование текста в текст.  
- **MoE** (Mixture of Experts) — модель, которая состоит из нескольких экспертов (предобученных моделей) для выполнения определённой задачи.

### Таблица 1.1 Сравнение архитектур больших языковых моделей

| Архитектура          | Применение                              | Пример модели     |
|----------------------|-----------------------------------------|-------------------|
| Только энкодер       | Классификация, извлечение признаков     | BERT, RoBERTa     |
| Только декодер       | Генерация текста                        | GPT, Llama        |
| Энкодер и декодер    | Машинный перевод, пересказ текста       | T5, BART          |
| Несколько экспертов  | В зависимости от имеющихся моделей      | Mistral           |

### Что такое Safetensors?  

- Ответ:  
  **Safetensors** — это безопасный и быстрый формат хранения весов нейросетей без исполнения кода, в отличие от pickle.
  Он защищает от RCE-уязвимостей и позволяет эффективно загружать большие модели (memory-mapped).

---

### Что хранится в файлах config.json и generation_config.json формата Safetensors?  

- Ответ:  
  В файле `config.json` формата Safetensors хранится метаинформация о модели, такая как архитектура, размер слоев, параметры обучения и другие настройки, необходимые для правильной загрузки и использования модели. Этот файл обеспечивает совместимость между различными версиями библиотек и помогает при восстановлении модели из весов.

  Пример содержимого `config.json` для модели LLaMA 2:
  ```json
  {
    "architectures": ["LlamaForCausalLM"], # Тип архитектуры модели
    "hidden_size": 4096, # Размерность эмбеддинга
    "intermediate_size": 11008, # Размер скрытого слоя в трансформере
    "num_attention_heads": 32, # Количество голов внимания
    "num_hidden_layers": 24, # Количество слоев трансформера
    "vocab_size": 32000, # Размер словаря токенов модели
    "max_position_embeddings": 2048, # Максимальная длина входной последовательности (контекст)
    "use_cache": true, # Использование KV-кэша для ускорения инференса
    "
    ...
  }
  ```

  Пример содержимого `generation_config.json` для модели LLaMA 2:
  ```json
  {
    "bos_token_id": 1, # ID токена начала последовательности
    "eos_token_id": 128009, # ID токена конца последовательности
    "max_length": 2048, # Максимальная длина генерируемой последовательности
    "max_new_tokens": 256, # Максимальное количество новых токенов для генерации
    "num_beams": 1, # Количество лучей для поиска (beam search)
    "temperature": 1.0, # Температура для управления разнообразием генерации (от 0.0 до 1.0)
    "top_k": 50, # Ограничение на количество наиболее вероятных токенов
    "top_p": 1.0, # Ограничение на кумулятивную вероятность токенов
    "repetition_penalty": 1.0, # Штраф за повторение токенов
    ...
  }
  ```

### Что такое GGUF?  

- Ответ:  
  **GGUF** - это формат файлов для хранения и запуска LLM, с самого рождения заточенный под работу **с квантованными моделями**, поддерживая множество градаций сжатия, таких как **Q4_K_M**, **Q5_K_S** и множество других форматов. Единственная цель GGUF — обеспечить максимально эффективный и экономичный инференс ИИ-моделей на доступном оборудовании, вплоть до обычных потребительских GPU и даже CPU.

---

### Что такое Квантизация?  
- Ответ:
  **Квантизация** - это уменьшение точности весов модели (например, с FP16 до INT8/INT4) для снижения потребления памяти и ускорения инференса ценой небольшой потери качества. Яркий пример: модель DeepSeek-R1 с 671 миллиардами параметров в исходном формате FP16 занимает около 404 ГБ, тогда как ее версия в GGUF с квантизацией Q4_K_M и 14 миллиардами параметров весит примерно 9 ГБ, что делает ее доступной для запуска на большинстве современных ПК. 

---

### Что такое Embeddings?  
- Ответ:  
  **Эмбеддинги (embeddings)** — это числовые представления слов, фраз или документов в виде векторов в многомерном пространстве. Они позволяют моделям машинного обучения понимать семантические связи между текстами, сравнивать их и выполнять задачи, такие как поиск по смыслу, кластеризация и классификация.

---

### Что такое тензор?  
- Ответ:  
  **Тензор** — это многомерный массив данных, который используется для представления и обработки информации в машинном обучении. Тензоры могут иметь различную размерность: скаляры (0D), векторы (1D), матрицы (2D) и более высокоразмерные структуры (3D и выше). Они являются основой для операций в нейронных сетях, позволяя эффективно хранить и манипулировать большими объемами данных.

### Что такое KV-Cache?  
  - Ответ: **KV-cache снижает сложность вычислений при генерации следующего токена с квадратичной O(n²) до линейной O(n)**,
где n — длина уже сгенерированного текста (текущий контекст).

  (хорошая статья)[https://serverflow.ru/blog/stati/chto-takoe-kv-cache-i-kak-on-uskoryaet-rabotu-yazykovykh-modeley-llm/] 

  **KV-кэш (Key-Value Cache)** — механизм оптимизации инференса LLM, сохраняющий промежуточные результаты вычислений (векторы Keys и Values) для всех обработанных токенов в памяти GPU. Это предотвращает избыточные вычисления на каждом шаге генерации, значительно снижая задержку (latency). Эффективное управление KV-кэшем (например, через PagedAttention в vLLM) является ключом к масштабированию систем с длинным контекстом.

  Более развернуто:

   **KV Cache** — это механизм кэширования промежуточных вычислений в архитектуре трансформеров, который сохраняет ключи (Key) и значения (Value) для всех предыдущих токенов во время инференса. Без KV Cache модель вынуждена была бы пересчитывать матрицы Key и Value для всей последовательности при генерации каждого нового токена, **что создавало бы квадратичную зависимость (O(n²)) между длиной контекста и требуемыми вычислениями**. **С использованием кэша сложность снижается до линейной (O(n))**, поскольку новые вычисления выполняются только для последнего токена, а для всех предыдущих используются закэшированные значения. Технически KV Cache хранит выходные значения ключей и значений для каждого слоя внимания и для каждой "головы" внимания в модели. При генерации очередного токена механизм внимания использует текущий токен как Query (запрос), а все Key и Value из кэша — для расчета весов внимания и итогового представления.  

  **KV Cache в vLLM**

**vLLM** внедрил одну из самых передовых реализаций — **PagedAttention**. Вместо того чтобы размещать KV-кеш непрерывно в памяти GPU, как это делалось ранее, **vLLM** разбивает кеш на страницы фиксированного размера. Это решает проблему фрагментации памяти и позволяет динамически перераспределять ресурсы между несколькими запросами. **Такое решение критично при выполнении инференса в нескольких пользовательских сессиях**. Поскольку страничный подход **поддерживает переиспользование выделенных блоков**, достигается прирост скорости до 24x относительно базовой реализации.

### Q (Query / Запрос), K (Key / Ключ), V (Value / Значение) в контексте внимания:

- Q (Query / Запрос): Что токен ищет в тексте.
- K (Key / Ключ): Что токен может предложить другим.
- V (Value / Значение): Сама информация, которую несет токен.

### Chunked Prefill
**Chunked Prefill** — это оптимизация процесса инференса в LLM, которая позволяет обрабатывать длинные входные последовательности, разбивая их на более мелкие части (чанки). Вместо того чтобы загружать весь контекст в память GPU сразу, модель обрабатывает его по частям, что снижает требования к VRAM и позволяет работать с более длинными текстами. Этот подход особенно полезен при использовании KV-кэша, так как он позволяет эффективно управлять памятью и поддерживать высокую производительность даже при работе с большими контекстами.

**Chunked prefill** решает три проблемы:

- Не даёт одному prefill «съесть» весь шаг (в батче на одном шаге GPU обрабатывает и prefill, и decode)
- KV резервируется порциями, а не разом
- Уменьшается latency для первых токенов на этапе decode, так как не ждём полного prefill, а запускаем параллельно сразу две очереди (prefill и decode)

### Что такое offloading (в контексте LLM)?

- Ответ: **Offloading** — это перенос части данных или вычислений с GPU на другую память или устройство (чаще всего CPU RAM или NVMe), чтобы уместить модель в ограниченную VRAM.

Проще:
> Не хватает видеопамяти → часть модели или KV-cache хранится вне GPU → данные подгружаются по мере необходимости.

### Retrieval-Augmented Generation (RAG)

**RAG (Retrieval-Augmented Generation)** — это технология, которая дает нейросети «внешнюю память» или «справочник», чтобы она перестала галлюцинировать и могла отвечать на основе свежих или закрытых данных.

![RAG_diagram](https://github.com/Rakhmankulov/Interviews/blob/main/images/RAG_diagram.jpg)

Если простая LLM — это **студент на экзамене**, который полагается только на свою память, то RAG — это **тот же студент, которому разрешили пользоваться библиотекой**.

### Как это работает (3 простых шага):

1.  **Поиск (Retrieval):** Когда вы задаете вопрос, система идет не сразу в нейросеть, а в специальную базу данных (обычно векторную). Там она ищет куски текста, которые по смыслу похожи на ваш запрос.
2.  **Дополнение (Augmented):** Найденные куски текста (контекст) приклеиваются к вашему вопросу. Получается длинный промпт: *«Используя вот эту информацию: [найденный текст], ответь на вопрос: [ваш вопрос]»*.
3.  **Генерация (Generation):** Нейросеть читает предоставленный текст и выдает точный ответ, опираясь на факты из «справочника».

### Почему это круто:
*   **Актуальность:** Не нужно переобучать модель, чтобы она узнала о сегодняшних новостях — достаточно обновить базу документов.
*   **Отсутствие галлюцинаций:** Модель ограничена вашим текстом. Если в базе нет ответа, она может сказать «я не нашел информации», а не выдумывать.
*   **Безопасность:** Можно скормить нейросети корпоративные документы, не отправляя их на дообучение в OpenAI (или любой другой внешней LLM-ке).
*   **Пруфы:** В ответе RAG-системы часто можно увидеть ссылку на конкретный документ, из которого взят факт.

### Из чего состоит RAG-стек:
*   **Векторная база данных** (Qdrant, ChromaDB, Pinecone, Milvus) — где лежат ваши знания.
*   **Эмбеддинги** (модели вроде `text-embedding-3`, `nomic-embed-text`) — превращают текст в математические векторы для поиска.
*   **Оркестратор** (LangChain, LlamaIndex) — связывает поиск и генерацию в один пайплайн.

### Что такое дрейф (drifting)

- Ответ:
  **Дрейф (дрифт)** — это когда **распределение данных или поведение модели меняется со временем**,
  а модель об этом «не знает».
  
  Важно:
  
  * модель **не ломается**
  * inference **отрабатывает**
  * метрики latency / error rate **зелёные**
  * а качество ответов — **уплыло**

  Бывает двух видов:  
1. **Data Drift** — когда входные данные со временем становятся другими (например, изменился стиль запросов пользователей).  
2. **Concept Drift** — когда сама задача или контекст меняются (например, модель обучалась на данных до 2020 года, а сейчас ей нужно отвечать на вопросы про события 2026 года).

### За какими метриками и параметрами модели нужно следить в продакшене? Как следить за качеством ответов и дрифтом?

Ответ: Мониторинг в продакшене делится на три уровня: **инфраструктурный** (здоровье железа), **инференс** (скорость ответов) и **семантический** (качество ответов).

#### 1. Технические метрики (Performance & Infra)
Это базовые показатели, которые говорят о том, насколько эффективно модель использует ресурсы.
- **TTFT (Time to First Token)**: Время до выдачи первого токена. Самая важная метрика для UX — если она > 1-2 сек, пользователь чувствует «тормоза».
- **Tokens Per Second (TPS)**: Общая пропускная способность.
- **TPOT (Time Per Output Token)**: Среднее время генерации каждого последующего токена.
- **VRAM KV-Cache Usage**: Заполненность кэша контекста. Если кэш забит, новые запросы встают в очередь (Queue Delay).
- **Model Latency vs Request Length**: Зависимость скорости от длины промпта (помогает выявлять аномалии в длинных контекстах).
- **CPU/GPU utilization, Memory/VRAM usage, Disk I/O**: Классические метрики для выявления узких мест в инфраструктуре.
- **Error rate (5xx, timeouts)**: Если растет, нужно срочно смотреть логи и состояние кластера.

#### 2. Мониторинг качества (Quality & Semantic)
LLM — недетерминированные системы, поэтому классические тесты тут не работают.
- **LLM-as-a-Judge**: Использование более мощной модели (например, GPT-4o) для оценки ответов вашей модели по шкале от 1 до 10.
- **Human feedback**: thumbs up / thumbs down - можно строить графики ответов пользователей и наблюдать за динамикой реакций.
- **Golden set**: Это аналог unit-тестов для модели:
  - Есть набор эталонных запросов
  - После выката новой модели — автоматический прогон
  - Сравниваем:
    - стабильность ответа
    - семантическую близость
    - hallucination rate    

- **RAGAS Metrics** (для RAG-систем):
  - **Faithfulness**: Насколько ответ соответствует найденному контексту (нет ли галлюцинаций).
  - **Answer Relevance**: Насколько ответ вообще соответствует вопросу.
  - **Context Precision**: Насколько качественные документы нашел ретривер.
- **Sentiment & Toxicity**: Следим, чтобы модель не начала хамить или выдавать запрещенный контент (используем Guardrails).

#### 3. Дрифт данных и ответов (Model & Data Drift)
Дрифт — это когда характеристики входных данных или ответов модели меняются со временем.
- **Embedding Drift**: Мы переводим ответы модели в векторы (эмбеддинги) и сравниваем их распределение с «эталонным» (из обучающей выборки или первой недели продакшена). Если среднее расстояние между векторами растет — модель начала «уплывать».
- **Prompt Drift**: Пользователи начали задавать вопросы, к которым модель не готова (например, на другом языке или в другом стиле).
- **Prediction Drift**: Если модель внезапно стала давать слишком короткие или слишком длинные ответы по сравнению с исторической нормой.

#### Как это реализовать (Инструменты):
- **Трейсинг**: 🍀**Langfuse, Arize Phoenix** или **Weights & Biases**. Они позволяют видеть дерево вызовов (промпт -> поиск -> ответ) и ставить оценки.
- **Сбор метрик**: 🍀**Prometheus + Grafana** (для vLLM/Triton есть готовые экспортеры).
- **Анализ дрифта**: **Evidently AI** (отличная библиотека для визуализации дрифта в ML/LLM).

### Форматы моделей в LLMOps:

*   **Safetensors:** Основной формат хранения весов. Безопасен, поддерживает быструю загрузку (mmap) и заменяет устаревший PyTorch Pickle.
*   **GGUF:** Оптимизирован для инференса на CPU/GPU (библиотека llama.cpp). Популярен для локальных LLM и edge-устройств.
*   **AWQ/GPTQ:** Форматы постобучающего квантования (4-bit). Позволяют кратно снизить требования к VRAM при минимальной потере качества.
*   **TensorRT:** Компилированные движки для максимальной производительности на GPU NVIDIA.
*   **FP8:** Современный стандарт для инференса на архитектурах Hopper (H100) и новее.

### Что такое eval в контексте LLM?

Ответ: **Eval** в контексте LLM — это оценка (evaluation) качества модели.

Это всё, что помогает понять, насколько хорошо модель отвечает на вопросы, следует инструкциям, не врёт, не повторяется и т.д.

- **Автоматический eval**  
  - Метрики: BLEU, ROUGE, BERTScore, Perplexity  
  - Судьи: другой LLM (GPT-4o, Claude, Llama-3.1-405B) сравнивает ответы  
  - Популярные бенчмарки: MMLU, HumanEval, MT-Bench, Arena-Hard, AlpacaEval

- **Человеческий eval**  
  - Люди читают и ставят баллы (Elo-рейтинг как в LMSYS Chatbot Arena)  
  - Самый точный, но дорогой и медленный

- **Production eval**  
  - Смотрят на реальные запросы пользователей  
  - Метрики: thumbs up/down, длина ответа, отказ отвечать, токсичность, галлюцинации  
  - Инструменты: LangFuse, Phoenix, TruLens, DeepEval

**Простыми словами:**  
Eval — это способ измерить, стала ли модель лучше после файнтьюна, промпт-инжиниринга или смены квантизации. Без нормального eval ты просто надеешься, что «кажется лучше».

### NVIDIA GPU Operator  
[оф дока](https://docs.nvidia.com/datacenter/cloud-native/gpu-operator/latest/index.html)  
Это инструмент для управления видеокартами NVIDIA внутри кластера Kubernetes. Разворачивается как Helm-Chart. 
Он сам обнаруживает новые узлы с GPU и подготавливает их к работе - устанавливает драйверы, настраивает библиотеки, ставит метки с информацией о картах на ноды и вводит их в эксплуатацию. А также, собирает метрики.

**Основные компоненты:**
1. **NVIDIA Driver**: Самый важный слой. Это программное обеспечение, которое объясняет операционной системе (и Kubernetes), как общаться с «железом» видеокарты.
2. **NVIDIA Container Toolkit**: Мостик между контейнерами и видеокартой. Позволяет приложениям внутри Docker-контейнеров «видеть» графический процессор и использовать его мощности.
3. **NVIDIA Device Plugin**: Регистрирует видеокарты как ресурсы в кластере. Благодаря ему Kubernetes понимает, что на узле есть, например, 4 свободные GPU, и может отправить туда ваши задачи.
4. **Node Feature Discovery (NFD)**: «Разведчик», который сканирует оборудование сервера и ставит на него метки (labels), сообщая оператору: «Эй, здесь есть видеокарта NVIDIA!».
5. **DCGM Exporter**: Сборщик метрик. Он вытягивает данные о температуре видеокарты, загрузке памяти и энергопотреблении, чтобы вы могли видеть красивые графики в Grafana.
6. **GFD (GPU Feature Discovery)**: Дополняет метки NFD более детальной информацией — например, сообщает конкретную модель видеокарты (A100, T4) и объем видеопамяти.
7. **MIG Manager**: Полезен для мощных карт (как A100 или H100). Позволяет «разрезать» одну физическую видеокарту на несколько независимых частей, чтобы их могли использовать разные пользователи одновременно.

---
### NVIDIA MIG Profiles (Multi-Instance GPU)
[оф дока](https://docs.nvidia.com/datacenter/tesla/mig-user-guide/getting-started-with-mig.html) 
Позволяет нарезать мощные и дорогие видеокарты на нужные кусочки и раздавать их инференс моделям разного размера, тем самым плотно утилизируя дорогое железо.

Разберем кодировку профилей, например, 1g.5gb:  
* **Цифра перед «g»** (1g, 2g, 3g...): Это количество «вычислительных долек» (GPU Slices). Всего у A100 их 7. Если ты берешь 1g, ты забираешь 1/7 мощности. Если 7g — забираешь всё.
* **Цифра после точки** (5gb, 10gb, 20gb...): Это объем выделенной видеопамяти.

**Важные ограничения из документации:**
*   **Фиксированные сетки (Placements):** Нарезка GPU ограничена физической структурой чипа; нельзя создать произвольную комбинацию профилей, если она не вписывается в поддерживаемую сетку размещения.
*   **Отсутствие динамического изменения:** Для смены профиля необходимо остановить текущие процессы и перенастроить инстанс.
*   **Поддержка архитектур:** Работает только на картах серий Ampere (A100, A30) и Hopper (H100).
*   **Только вычисления:** MIG разработан для CUDA-задач и не поддерживает графические API (OpenGL, Vulkan).
*   **Изоляция:** Обеспечивает полную аппаратную изоляцию (QoS), гарантируя, что нагрузка в одном инстансе не влияет на производительность другого.

### Обучение и дообучение

#### Что такое SFT?
**SFT (Supervised Fine-Tuning)** — это контролируемое дообучение языковой модели на заранее подготовленном наборе качественных данных формата «запрос — идеальный ответ». Цель SFT — улучшить способность модели следовать инструкциям и выдавать более релевантные ответы, не изменяя её базовые знания.

#### Что такое RLHF?
**RLHF (Reinforcement Learning from Human Feedback)** — это метод обучения нейросетей, при котором модель дорабатывается на основе оценок и предпочтений человека.

#### Что такое Lora?
**LoRA (Low-Rank Adaptation)** — это метод эффективного дообучения больших языковых моделей (LLM). Вместо изменения всех миллиардов параметров исходной модели, он замораживает ее и обучает лишь крошечную часть дополнительных весов, что позволяет запускать процесс на обычном оборудовании.

### Inference runtime серверы

**Inference runtime сервер** — это сервис, который загружает ML/LLM-модель в память (CPU/GPU) и предоставляет API для выполнения инференса (generations, embeddings, chat и т.п.).
Он отвечает за:

* загрузку и хранение модели в памяти,
* эффективное использование GPU/CPU,
* батчинг и параллелизм запросов,
* управление контекстом и KV-cache,
* API (чаще всего OpenAI-compatible).

Используется как backend для приложений, RAG, агентов и внутренних сервисов.

### Ollama

**Ollama** [оф дока](https://docs.ollama.com/quickstart) — простой inference runtime, ориентированный на локальный запуск моделей.

#### Ключевые особенности

* Минимум настроек
* Поддержка CPU и GPU

#### Плюсы

#### Типовые use-cases

* Локальная разработка
* Демонстрации
* Тестирование моделей
* Небольшие внутренние сервисы

### vLLM

Q: Использовали квантизацию?
A: Да. Квантизация влияла на скорость генерации и потребление GPU. Чем больше квантизация (например, 4 бит вместо 16), тем быстрее работает модель и меньше VRAM она занимает, но при этом может немного падать качество ответов (2-3% при INT8). В продакшене мы обычно выбираем компромисс между производительностью и качеством, ориентируясь на конкретные задачи и требования пользователей.

Q: Как квантизация влияет на качество?
A: Например, переход с 16 на 8 бит у Mistral давал значительный прирост скорости при допустимом падении качества (2-3%) для конкретной задачи.

### Какую платформу или фреймворк можно использовать для сервинга моделей?

- Ответ:

  **Самые частые комбинации в 2026 (production):**
  - **Быстро и просто** → BentoML + Docker/K8s или BentoML + LiteLLM Proxy
  - **LLM на максимальной скорости** → vLLM (или TGI) внутри KServe / Seldon
  - **Kubernetes-стандарт** → KServe + vLLM / Triton runtime
  - **Без K8s и без головной боли** → Hugging Face Inference Endpoints или cloud-managed (SageMaker / Vertex)
  
  Если у тебя Kubernetes → начинай с **KServe** (самый современный и лёгкий).  
  Если LLM → **vLLM** + BentoML/KServe.  
  Если классический ML и хочешь минимум кода → **BentoML**.

### Для чего используется NVIDIA Device Plugin в Kubernetes?

- Ответ:

  **NVIDIA Device Plugin** в Kubernetes — это DaemonSet, который **делает GPU видимыми и доступными** для подов.
  
  Коротко, для чего он нужен:
  
  - **Регистрирует** все GPU на ноде как ресурс `nvidia.com/gpu` (например, `nvidia.com/gpu: 1` в Pod spec).
  - Позволяет **Kubernetes scheduler** автоматически размещать поды на нодах с нужным количеством GPU.
  - Передаёт GPU в контейнер (через env `NVIDIA_VISIBLE_DEVICES`, CDI или другие способы).
  - Поддерживает **MIG** (Multi-Instance GPU), time-slicing, MPS для шаринга GPU.
  - Мониторит здоровье GPU и сообщает kubelet'у.
  
  Без него Kubernetes просто не знает про GPU — поды не получат доступ, даже если драйверы и   nvidia-container-toolkit стоят.
